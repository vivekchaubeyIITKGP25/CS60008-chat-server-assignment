## 1. Scope and Approach

I implemented a thread-based chat system in Python using `socket` and `threading`, then extended it with authentication, duplicate-login policy, rooms, publish-subscribe messaging, Redis-backed shared state, TLS transport, and Docker-based deployment.

The project goal was to satisfy all eight assignment problems while keeping the code simple to run and inspect. I focused on a straightforward architecture:

- one process per server instance
- one thread per connected client
- Redis as the shared state and cross-server message bus
- thin command-line client for interactive testing

This approach keeps the control flow explicit and matches the assignment requirement to understand blocking I/O and thread lifecycle behavior.

## 2. Design Choices

### 2.1 Thread model

I used a dedicated thread for each accepted client connection (`threading.Thread`). The server loop only handles `accept()`, and per-client threads run authentication and message command loops. This makes failure isolation simple: one bad client does not block other clients.

### 2.2 Session policy for duplicate logins

For Problem 3, I chose **Force Logout Existing Session**.

Behavior:
- if `LOGIN username password` is received and a session already exists,
- the old client is notified first,
- the old session is disconnected and cleaned,
- the new login is accepted.

This policy is documented in code and README. I also scoped cross-server force-logout notifications so that only the server holding the old session performs the disconnection.

### 2.3 Messaging model

I implemented two messaging paths:

1. Room mode: user has a non-empty current room; messages go to members of that room.
2. Publish-subscribe mode: user leaves room (`/leave`), then publishes to subscribers only.

The dual-path model satisfies both Problem 4 and Problem 5 while keeping user commands explicit.

## 3. Redis Integration and Schema

Redis is used as the single source of truth for global runtime state.

### 3.1 Keys and structures

- `user:<username>` (hash)
  - `password` (bcrypt hash)
  - `created` (timestamp)

- `session:<username>` (hash)
  - `username`
  - `server_id`
  - `room`
  - `login_time`
  - `session_id`

- `room:<room_name>` (set)
  - usernames currently in that room

- `subscribers:<publisher_username>` (set)
  - usernames subscribed to the publisher

### 3.2 Cross-server behavior

Each server instance subscribes to `chat_broadcast` (Redis Pub/Sub).  
When a client sends a room or pub-sub message, the local server publishes a JSON payload to this channel. Every server receives it, checks only its local sockets, and delivers to matching recipients.

This makes the system stateless regarding global room/session membership and satisfies the requirement that users connected to different server instances still receive same-room messages.

## 4. Thread Safety and Concurrency Notes

### 4.1 Local shared state

The only mutable in-process shared structure is the local socket map (`clients`). I protect it using `clients_lock` during add/remove/iteration operations. This avoids races while broadcasting or disconnecting users.

### 4.2 Redis-side consistency

Most shared updates are performed via atomic Redis commands (`HSET`, `SADD`, `SREM`, `DEL`). This helps avoid in-process locking for global state.

### 4.3 Important race conditions handled

- **Duplicate login across servers**: old session is notified and terminated before new session is finalized.
- **Late messages from stale sockets**: if Redis session no longer exists, server returns a session-expired error and disconnects that flow.
- **Disconnect cleanup**: on thread exit, room membership and session keys are removed when ownership checks match current session id.

### 4.4 Known limitations

- No durable message history.
- No strict transactional grouping across multiple Redis keys.
- Uses Redis `KEYS room:*` and `KEYS subscribers:*` in command handlers, which is acceptable for assignment scale but not ideal for large production deployments.

## 5. TLS Decisions

I used Python `ssl` for encrypted transport.

Server:
- wraps listening socket with `ssl.PROTOCOL_TLS_SERVER`
- loads `server.crt` and `server.key`
- accepts only TLS clients when TLS is enabled

Client:
- uses `ssl.PROTOCOL_TLS_CLIENT`
- sets `CERT_REQUIRED`
- verifies server certificate using pinned local cert (`server.crt`)

For local assignment testing, I used self-signed certificates generated by `generate_cert.sh`. This provides confidentiality and certificate verification without external CA dependency.
